Observation Project Owner Guide

Observation 프로젝트를 책임지기 위한 전체 가이드

이 문서는 Spring Boot starter-first observability dashboard 프로젝트를 처음 보는 사람이 구조, 용어, 데이터 흐름, 세부 판정 로직, API, 운영 제약을 설명할 수 있도록 정리한 오너용 기준 문서다. 기준일은 2026-06-01이며, 기존 기획 문서와 현재 구현을 함께 확인해 작성했다.

주요 제품
Spring Boot starter가 30초 bucket과 heartbeat를 포털로 보내고, 포털이 dashboard read model을 만든다.
현재 아키텍처
Traditional MVC + Service/Repository Layering. 순수 DDD/Hexagonal이 아니다.
원천 DB
PostgreSQL + Flyway SQL migration. JPA DDL 자동 생성은 꺼져 있다.
보안 원칙
GitHub OAuth account auth, Bearer service JWT, starter credential raw value 1회 표시.

1. Owner Summary

오너가 먼저 잡아야 할 핵심

이 프로젝트를 책임진다는 것은 “포털 화면이 왜 이 상태를 보여주는지”를 accepted bucket, heartbeat, snapshot, state rule, endpoint evidence까지 역추적할 수 있다는 뜻이다. 특히 heartbeat는 control-plane 연결 신호이고, accepted metric bucket은 data-plane 관측 신호다. 두 신호를 섞어서 host application의 실제 생존 여부를 단정하면 안 된다.

읽기 모델의 주인

Dashboard 상태, zeroInsight, triage card, endpoint priority는 서버 service가 계산한다. UI, controller, repository는 이 의미를 다시 만들지 않는다.

저장 원칙

PostgreSQL은 accepted 30초 bucket, heartbeat 최신 상태, snapshot read model을 저장한다. Prometheus나 TSDB, 임의 query explorer는 MVP 경로가 아니다.

Percentile 원칙

p95/p99는 starter가 보낸 summary.localPercentiles의 starter_local, instance_bucket point만 canonical source다. 포털은 histogram으로 p95/p99를 재계산하지 않는다.

Snapshot 원칙

snapshot/history/trend/detail은 저장된 dashboard_snapshots.read_model_json을 projection한다. 현재 dashboard나 bucket을 다시 join해 과거를 재계산하지 않는다.

Starter 원칙

host request path에서는 네트워크 호출을 하지 않는다. Micrometer observation은 로컬 guard, rollup, bounded queue까지만 지나고, background worker가 flush한다.

보안 원칙

project key, refresh token, OAuth state nonce 원문은 저장하지 않는다. raw starter credential은 생성/회전 성공 응답에서만 1회 표시한다.

현재 구현과 계약 문서의 중요한 차이

주제 문서/의도 현재 구현 기준 오너 판단
Ingest idempotency 같은 payload의 duplicate는 200으로 볼 여지가 문서에 남아 있다. IngestAcceptanceService는 같은 project의 기존 Idempotency-Key를 발견하면 payload hash 비교 없이 409로 수렴한다. 운영/테스트 기대값은 409로 잡아야 한다. 200 재처리를 원하면 service와 contract를 함께 바꿔야 한다.
Retention accepted bucket 7일, snapshot 14일 horizon이 계약에 있다. snapshot query/scheduler는 portal.dashboard-snapshots.retention-days로 clamp한다. 현재 cleanup package는 예약만 되어 있고 실제 삭제 job은 보이지 않는다. 저장소 용량 정책을 운영 요구로 삼으려면 cleanup story가 필요하다.
Docs 위치 _bmad config의 project knowledge는 docs를 가리킨다. 작성 전 docs/ 폴더는 없었고, 실제 원천은 planning-artifacts, implementation-artifacts, _bmad/custom/project-context.md였다. 이 파일을 새 기준 문서로 두고 이후 docs 체계를 맞추는 것이 좋다.

3. Product Model

제품을 이해하는 한 장짜리 모델

Spring Boot host app
  -> observability-spring-boot-starter
  -> 30초 accepted metric bucket + 30초 heartbeat
  -> observability-portal
  -> PostgreSQL read-side persistence
  -> Application Dashboard
  -> Snapshot / History / Instance Evidence drill-down

Scope 계층

제품의 계층은 project -> application -> instance다. Project Entry와 Application List는 scope 선택 화면이고, 운영 판단의 primary first screen은 Application Dashboard다. Instance Evidence와 Snapshot Trend는 근거 drill-down이다.

두 데이터 축

accepted metric bucket은 실제 traffic 관측 data-plane이다. heartbeat는 starter가 살아 있고 포털까지 도달했다는 control-plane telemetry다. dashboard는 둘을 나란히 표시하지만 state freshness 계산은 accepted bucket 기준이다.

이 프로젝트가 하지 않는 일

  • Prometheus scrape, TSDB, arbitrary query UI를 제공하지 않는다.
  • 포털에서 histogram을 합쳐 p95/p99 scalar를 새로 만들지 않는다.
  • heartbeat만으로 host application이 정상/다운이라고 단정하지 않는다.
  • email/password, cookie session, GitHub provider token 저장을 만들지 않는다.
  • route attribution에 raw URL, query string, high-cardinality tag를 사용하지 않는다.
  • archive/의 legacy 구현을 현재 기준으로 되살리지 않는다.

4. Glossary

프로젝트 용어 사전

용어 의미 주의점
Project starter credential과 application 묶음의 최상위 scope. DB table은 projects. account membership으로 접근 제어한다. starter credential status와 project status는 분리되어 있다.
Starter Credential <key_prefix>.<secret> 형태의 raw project key. starter가 X-OBS-Project-Key로 보낸다. 원문은 저장하지 않고 BCrypt hash만 저장한다. 생성/회전 시 한 번만 보여준다.
Application project 안의 application.name + environment 조합. heartbeat만으로 catalog를 만들지 않는다. accepted bucket 수용 시 catalog upsert된다.
Instance application 안의 runtime instance identity. instance evidence는 selected instance scope를 보여주지만 application triage와 함께 읽는다.
Accepted Metric Bucket 검증을 통과해 포털 DB에 저장된 30초 UTC metric 집계. dashboard state의 freshness source다. heartbeat와 별도다.
Heartbeat starter가 주기적으로 보내는 control-plane telemetry. catalog upsert나 accepted bucket 저장을 하지 않는다. 최신 row만 upsert된다.
Dashboard Read Model application dashboard 화면이 소비하는 서버 계산 응답. UI가 state, p95/p99, endpoint priority를 재계산하면 안 된다.
Source-scoped Percentile starter-local instance bucket 단위 p95/p99 point. source=starter_local, scope=instance_bucket, mergeable=false만 유효하다.
Histogram Distribution duration cumulative histogram bucket 분포 evidence. boundary set이 window 안에서 모두 같을 때만 합산한다. percentile 계산용이 아니다.
Endpoint Key METHOD + " " + normalizedRoute. UNKNOWN route는 endpoint priority에서 제외된다.
Freshness 마지막 accepted bucket endUtc 기준 current/stale/down 후보. 90초 이상 stale, 180초 이상 down 후보. 첫 데이터가 없으면 waiting_first_data.
Lifecycle State waiting_first_data, unknown, idle, active, stale, down, degraded. metric axis와 starter connection axis를 분리해서 결정한다.
Zero Insight triage card가 없을 때 UI가 보여주는 “지금 왜 조치가 없는지” 설명. 상태를 새로 판단하는 기능이 아니라 서버 reasonCode 표시다.
Triage Card application-level first-check suggestion. root cause engine이 아니다. 최대 3개, confidence 0.65 이상만 노출한다.
Endpoint Priority 우선 확인할 endpoint 후보 ranking. current freshness와 sample sufficient 조건에서만 계산하며 최대 5개다.
Snapshot 특정 시점 dashboard read model JSON을 저장한 coarse-grained row. history/detail/trend의 source다. 과거 상태를 live data로 재계산하지 않는다.
Operational Event stored snapshot sequence에서 projection한 사건 목록. 별도 테이블이 없다. suppression/folding은 projection 단계에서 이뤄진다.

5. Modules

모듈과 배포 단위

6. Architecture Boundaries

아키텍처와 패키지 경계

현재 구현 기준은 Traditional MVC + Service/Repository Layering이다. domain은 DDD layer가 아니라 feature namespace다. application, port, adapter 같은 Hexagonal package를 새로 만들지 않는 것이 현재 결정이다.

Layer/Package 책임 하면 안 되는 일
controller HTTP boundary, status mapping, request/response DTO 변환. state/rule/priority를 계산하지 않는다.
service use case orchestration, validation, read model 조립, 정책 판단. raw secret을 로그/response에 누출하지 않는다.
repository JPA/native query, DB projection, insert/upsert. lifecycle state, insight rule, endpoint priority를 계산하지 않는다.
model/dto service 내부 값, API response shape, contract object. 비밀 원문이나 high-cardinality raw path를 보존하지 않는다.
entity DB row mapping. Flyway schema와 어긋나는 필드/제약을 임의 추가하지 않는다.

포털 feature namespace

주요 namespace는 account, catalog, ingest, bucket, dashboard, instance, snapshot, history, state, common.time이다. cleanup namespace는 예약되어 있지만 현재 실질 cleanup service는 없다.

7. Data Model

DB schema와 저장 의미

물리 schema는 observability-portal/src/main/resources/db/migration의 Flyway SQL이 원천이다. spring.jpa.hibernate.ddl-auto=none이므로 JPA entity가 schema를 자동 생성하지 않는다.

Table 핵심 컬럼 의미와 제약
projects name, key_prefix, project_key_hash, status, starter credential metadata. project name/key prefix/hash는 unique. status는 active/disabled. starter credential은 active/revoked.
applications project_id, name, environment, first_seen_at, last_seen_at. project_id + name + environment unique. accepted ingest가 catalog source다.
application_instances application_id, instance_name, first_seen_at, last_seen_at. application_id + instance_name unique. dashboard instance list는 lastSeen desc/name asc, 최대 50개.
accepted_metric_buckets schema/version, idempotency key, payload hash, bucket time, counts, histogram JSON, local percentile JSON, endpoint JSON, runtime ratios. schema 1.0, duration 30초, count/range JSON check, project_id + idempotency_key unique, application_instance_id + bucket_start_utc unique.
starter_heartbeat_telemetry project/application/env/instance, starter version, last sent/received, sequence, interval, status. identity별 최신 heartbeat row를 upsert. catalog source가 아니다.
dashboard_snapshots project/application, generated/window times, state code, capture reason, helper columns, read_model_json. application_id + current_window_end_utc unique. writer가 priority-aware upsert한다.
accounts, external_identities internal account와 GitHub provider subject mapping. GitHub OAuth only. email은 stable identity key가 아니다.
refresh_token_families, refresh_tokens family status, token hash, expiration, consumed/revoked/reuse timestamps. refresh token 원문 저장 금지. reuse detection 시 family까지 reuse_detected로 닫는다.
oauth_state_nonces nonce hash, status, expires/consumed. OAuth state replay 방지. nonce 원문은 저장하지 않는다.
account_project_memberships account_id, project_id, role, status. MVP role은 member만 허용. project-scoped API authorization source다.

8. API Surface

API 표면과 인증 경계

공개 ingest API: starter credential 사용

Endpoint Auth 성공 실패 의미
POST /api/ingest/v1/buckets X-OBS-Project-Key, Idempotency-Key 201 Created 400 invalid, 401 unauthorized, 409 duplicate 검증된 30초 metric bucket을 저장하고 catalog를 upsert한다.
POST /api/ingest/v1/heartbeat X-OBS-Project-Key 200 received 400 invalid, 401 unauthorized starter heartbeat 최신 telemetry를 upsert한다. bucket/catalog/state 계산은 하지 않는다.

Account auth API: 공개 JSON boundary

Endpoint 의미 주의
GET /api/auth/github/authorize signed OAuth state를 만들고 GitHub authorize URL을 JSON으로 반환한다. no-store header. server session/cookie를 만들지 않는다.
GET /api/auth/github/callback state nonce를 소비하고 GitHub code를 교환한 뒤 service token pair를 반환한다. GitHub provider token은 user lookup 뒤 저장하지 않는다.
POST /api/auth/token/refresh refresh token rotation. 새 access/refresh pair를 반환한다. 기존 token은 rotated로 소비된다. 재사용 감지 시 family를 reuse_detected로 닫는다.
POST /api/auth/logout refresh token과 family를 revoke한다. 204 No Content.

Resource API: Bearer service access token 사용

Endpoint Scope Authorization
GET /api/projects account가 active membership을 가진 project 목록. Bearer access token 필요.
POST /api/projects project 등록과 starter credential 1회 발급. Bearer access token 필요.
GET /api/projects/{projectId}/applications project application list. Bearer + active membership. mismatch/invalid는 body 없는 404.
GET /api/projects/{projectId}/applications/{applicationId}/dashboard application dashboard current read model. project/application mismatch는 404.
GET /api/projects/{projectId}/applications/{applicationId}/instances/{instanceId}/evidence selected instance current evidence. UUID path 정합성 mismatch는 404.
GET /api/projects/{projectId}/applications/{applicationId}/instances/{instanceId}/snapshot-trend stored snapshots에서 selected instance trend projection. since=7d|14d, limit 기본 168, 최대 336.
GET /api/projects/{projectId}/applications/{applicationId}/dashboard/snapshot-markers stored snapshot marker timeline. since=24h|7d|14d, retention과 최대 14일로 clamp.
GET /api/projects/{projectId}/applications/{applicationId}/dashboard/snapshots/{snapshotId} stored snapshot detail. invalid UUID 400, missing/expired/path mismatch 404.
GET /api/projects/{projectId}/applications/{applicationId}/operational-events stored snapshot 기반 operational event history. since는 양의 정수 + h/d, 최대 14일. 별도 event table은 없다.
GET/POST /api/projects/{projectId}/starter-credential... starter credential metadata/rotation/revocation. metadata는 raw/hash를 반환하지 않는다. rotation만 raw value 1회 반환.

9. End-to-End Flows

핵심 흐름

9.1 Account와 project onboarding

1
GitHub authorize 시작
포털은 signed state를 만들고 nonce hash를 DB에 저장한다.
2
Callback 완료
state 서명/만료/nonce를 검증하고 nonce를 consumed 처리한 뒤 GitHub code를 token으로 교환한다.
3
Account 연결
GitHub provider subject로 기존 identity를 찾거나 account/external_identity를 만든다.
4
Service token 발급
HS256 access JWT와 32 random byte refresh token을 발급한다. refresh token은 SHA-256 hash만 저장한다.
5
Project 등록
project name을 lower-case 정규화하고 credential을 생성해 project와 active membership을 함께 만든다.

9.2 Starter metric bucket ingest

1
Micrometer observation
MicrometerHttpServerObservationBinder가 HTTP server observation의 method/status/error/duration/route 후보를 추출한다.
2
Route guard
LowCardinalityHttpObservationGuard가 raw/high-cardinality 후보를 제거하고 normalized route 또는 UNKNOWN으로 수렴한다.
3
30초 rollup
MetricBucketRollupService가 app/endpoint count, cumulative histogram, local percentile, latest runtime ratio를 bucket별로 모은다.
4
Drain과 queue
bucket end + 30초 grace가 지난 bucket만 sealed snapshot으로 drain되고 bounded queue에 non-blocking enqueue된다.
5
Background flush
MetricBucketFlushWorker가 envelope와 idempotency key를 만들고 JDK HTTP client로 포털 ingest API에 보낸다.
6
Portal acceptance
포털은 project key, idempotency key, payload shape, route/count/histogram/localPercentiles를 재검증하고 DB에 저장한다.

9.3 Heartbeat ingest

Heartbeat는 metric bucket과 완전히 별도 흐름이다. starter의 daemon sender가 interval마다 schemaVersion, starterVersion, sentAtUtc, sequence, application identity를 보낸다. 포털은 project key와 metadata를 검증하고 starter_heartbeat_telemetry를 upsert한다. response에는 마지막 accepted bucket 시각을 참고 정보로 담을 수 있지만, heartbeat가 catalog를 만들거나 lifecycle state를 직접 바꾸지는 않는다.

9.4 Dashboard read model

1
Scope 검증
application이 project에 속하는지 확인한다. mismatch는 404.
2
시간 계산
query time을 30초 boundary로 floor한 evaluationAt을 기준으로 current 15분과 baseline 15분을 만든다.
3
Evidence 조회
latest bucket, aggregate, percentile rows, histogram rows, endpoint rows, gap evidence, recent 5 buckets, runtime ratio, heartbeat를 조회한다.
4
서버 계산
freshness, sample readiness, lifecycle state, starter connection, source-scoped percentile, histogram distribution, triage, endpoint priority를 조립한다.
5
Fallback snapshot
dashboard query 후 최근 snapshot이 부족하면 query fallback capture를 시도하되, 실패는 화면 응답을 막지 않는다.

9.5 Snapshot/history/trend

UTC 정시에 scheduler가 active application 중 retention horizon 안 accepted bucket이 있는 대상을 찾고 hourly snapshot capture를 dispatch한다. writer는 같은 application_id + current_window_end_utc에 대해 대표 reason priority가 더 높은 경우만 update하고, 같거나 낮으면 no-op한다. Detail, marker, instance trend, operational events는 저장된 snapshot JSON과 helper column만 사용한다.

10. Detailed Logic

세부 로직을 책임질 때 알아야 할 것

10.1 Project key 검증

  • Header는 X-OBS-Project-Key이고 형식은 <prefix>.<secret>다.
  • prefix는 32자 이하, 전체 raw key는 BCrypt 입력 한계 때문에 UTF-8 72 bytes 이하여야 한다.
  • repository lookup은 prefix로 하고, BCrypt hash match가 성공해야 verified project가 된다.
  • project가 disabled이거나 starter credential이 revoked면 unauthorized다.
  • 검증 결과 모델에는 raw key가 남지 않는다.

10.2 Ingest validation

대상 검증
schema schemaVersion=1.0만 허용.
application identity name/environment/instance는 blank, 앞뒤 공백, control character를 거부.
bucket start/end는 ISO-8601 UTC instant, 30초 boundary, duration 30초, end가 start보다 뒤.
counts request/error count는 음수 금지, errorCount는 requestCount 이하.
histogram 비어 있으면 invalid. leMs는 양수 증가, count는 cumulative이고 requestCount 이하.
runtime ratio CPU/heap/datasource pool ratio는 있으면 0.0 이상 1.0 이하.
localPercentiles scope/source/bucket boundary/requestCount/p95/p99/mergeable을 모두 검증. p99는 p95 이상.
endpoints method allowlist, normalized route 안전성, count/histogram 검증.
Idempotency-Key 5 components: project/application/environment/instance/bucketStart. application/env/instance/bucketStart는 payload와 일치해야 한다.

10.3 Route attribution

  1. http.route low-cardinality framework route가 있으면 먼저 사용한다.
  2. safe route template이면 normalize한다. ?... omission marker와 안전한 /... collapse marker만 허용한다.
  3. template syntax가 있지만 malformed이면 UNKNOWN으로 수렴한다.
  4. framework route가 실패하면 low-cardinality uri 또는 path 후보를 query 제거 후 임시 사용한다.
  5. raw 후보는 allowlist exact-one match를 먼저 시도한다. ambiguous면 UNKNOWN.
  6. allowlist match가 없으면 safe prefix collapse를 시도한다.
  7. http.url, high-cardinality tag, query string, absolute URL, concrete id-looking segment는 사용하지 않는다.

10.4 Starter rollup과 flush

  • 기본 histogram upper bound는 50, 100, 250, 500, 1000 ms.
  • local percentile은 instance bucket 전체 duration sample에서 nearest-rank 방식으로 p95/p99를 계산한다.
  • bucket duration은 30초이고 grace도 30초다. bucket.endUtc + 30s <= now일 때 drain된다.
  • drain된 interval은 sealed가 되고 이후 늦게 들어온 sample은 late drop count만 증가한다.
  • queue capacity 기본값은 1024, drop policy 기본값은 DROP_NEWEST.
  • DROP_OLDEST를 쓰면 oldest poll 후 새 bucket enqueue를 시도한다.
  • flush worker retry 기본값은 최대 3회, backoff 100ms.
  • metric flush portal connection setting은 observation.metric-flush.portal-base-url과 project-key가 둘 다 있어야 한다.
  • 실제 portal flush worker가 뜰 때 generic identity local-project, application, default, local-instance는 거부된다.

항목 값 의미

10.7 Zero insight precedence

triage card가 하나라도 있으면 zeroInsight는 null이다. triage card가 없을 때 reason은 아래 우선순위로 선택된다.

  1. recovery 관찰 중이면 observing_recovery.
  2. first data 대기 + recent heartbeat면 waiting_first_data.
  3. first data 대기 + heartbeat 없음/오래됨이면 telemetry_unreachable.
  4. stale/down + recent heartbeat면 metric_data_idle.
  5. stale/down + heartbeat 없음/오래됨이면 telemetry_unreachable.
  6. sample insufficient면 insufficient_sample.
  7. traffic idle이면 metric_data_idle.
  8. 나머지는 no_action_needed.

10.8 Triage rule thresholds

Rule Guard 노출/상태 영향
Global error spike current/baseline request >= 30, current error rate >= 5%, delta >= 3%p, relative >= 2x. confidence >= 0.65이면 card. degraded candidate.
Global latency spike current/baseline request >= 30, 같은 histogram boundary set, 500ms 초과 share >= 20%, delta >= 10%p. confidence >= 0.65이면 card. degraded candidate.
DB pool high with latency latency candidate + datasource pool usage >= 0.85. INFO hint. degraded candidate는 아니다.
CPU high with latency latency candidate + CPU usage >= 0.85. INFO hint.
Heap high hint latency 또는 error candidate + heap usage >= 0.90. INFO hint.

10.9 Endpoint priority

  • application aggregate sample readiness가 insufficient이면 전체 endpoint priority는 빈 배열이다.
  • freshness가 current가 아니면 빈 배열이다.
  • current/baseline endpoint request count가 각각 30 이상이어야 main spike 후보가 된다.
  • error spike와 latency spike threshold는 global rule과 같다.
  • comparative regression은 current/baseline request >= 100, error delta >= 2%p 또는 slow share delta >= 8%p이며 confidence cap은 0.64다.
  • reason priority는 ERROR_AND_LATENCY, ERROR_SPIKE, LATENCY_SPIKE, COMPARATIVE_REGRESSION 순이다.
  • 최대 5개만 반환한다.

10.10 Snapshot capture reason priority

Reason Token Priority 조건

10.11 Operational event projection

  • source는 dashboard_snapshots row metadata/helper column/read model JSON뿐이다.
  • state transition event는 degraded/stale/down/recovery/state_changed로 projection된다.
  • high-confidence concern threshold는 0.82다.
  • concern resolution threshold는 0.60이다.
  • 같은 concern은 60분 suppression window 안에서 반복 promotion하지 않는다.
  • 반복 period event는 folding key로 접는다.

11. Static Dashboard

정적 dashboard UI가 하는 일과 하지 않는 일

UI 파일은 observability-portal/src/main/resources/static/dashboard 아래에 있다. index.html은 Project Entry, Application List, Dashboard, Setup Guide를 한 화면에 배치한다. app.js는 API 응답을 표시하고, styles.css는 UI 스타일을 담당한다.

UI가 하는 일

  • GitHub authorize URL을 호출해 로그인 시작.
  • service access token을 메모리에 보관하고 resource API에 Bearer header로 전달.
  • project/application/dashboard/instance/snapshot/history API 응답을 표시.
  • credential rotation 성공 시 raw value를 1회 표시.

UI가 하지 않는 일

  • lifecycle state를 다시 계산하지 않는다.
  • p95/p99를 계산하지 않는다.
  • endpoint priority와 triage confidence를 만들지 않는다.
  • secret을 local storage에 저장하도록 요구하지 않는다.
