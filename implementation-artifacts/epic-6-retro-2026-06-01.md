# Epic 6 Retrospective - Dashboard User Flow and Demo Hardening

## Epic Review

Epic 6은 Epic 5에서 닫은 server-computed read model/API를 사용자가 실제로 밟는 화면 흐름으로 연결하는 epic이었다. 회고 기준은 `implementation-artifacts/sprint-status.yaml`, `implementation-artifacts/epic-6-context.md`, Epic 6 story files의 Dev Agent Record와 Completion Notes, Epic 5 retrospective action items, 최신 product source-of-truth와 contracts다.

현재 closure 판단은 가능하다. `sprint-status.yaml` 기준 Story 6.1부터 6.10까지 모두 `done`이고, 본 회고에서 `epic-6-retrospective`와 `epic-6`도 `done`으로 전환한다.

Epic 6에서 닫은 핵심 흐름은 아래와 같다.

- Account entry와 Project Entry는 GitHub OAuth only, Bearer service token, setup guide, project scope 선택으로 제한했다.
- Project Selection과 Application List는 scope를 좁히는 화면이며 Application Dashboard 판단을 앞당겨 대체하지 않았다.
- Application Dashboard는 Epic 5 dashboard read model을 표시하는 primary first-screen으로 구현했다.
- Instance Evidence, Instance Snapshot Trend, Snapshot/History는 각각 evidence drill-down, stored snapshot projection, bounded history/deep link로 유지했다.
- Green path와 failure/recovery demo는 safe copy, fail-closed guard, source-axis separation, membership boundary를 검증하는 hardening 흐름으로 닫았다.
- Account-project membership은 authenticated account의 active membership 안에서 project visibility와 project-scoped resource API를 제한하도록 닫았다.

## 잘 된 점

| 주제 | 관찰 | 반복 가치 |
|---|---|---|
| Story 순서가 사용자 흐름을 따랐다 | Project Entry에서 Application List, Dashboard, Instance Evidence, Trend/History, demo hardening으로 이어졌다. | 후속 UI도 사용자가 밟는 순서대로 API 소비와 guard를 확장하는 방식이 안전하다. |
| Negative guard가 구현을 보호했다 | no UI recomputation, no token persistence, no frontend build stack, no raw explorer, no project creation guard가 반복됐다. | "만들지 않은 것"을 테스트와 review checklist에 남기는 방식은 계속 유지한다. |
| Fail-closed UI pattern이 자리 잡았다 | link shape, selected context, response identity, malformed payload, stale response, membership mismatch가 안전한 상태로 수렴했다. | 외부로 보이는 UI가 늘어날수록 identity와 source validation을 먼저 둔다. |
| Copy guard가 계약을 사용자 언어로 연결했다 | stale/down, telemetry unreachable, metric idle, recovery observed, empty history가 단정 표현으로 번역되지 않게 했다. | 운영 화면에서는 계산 경계만큼 문구 경계도 source-of-truth다. |
| Membership boundary를 흐름 안에 통합했다 | Story 6.10은 Project Entry 흐름을 다시 설계하지 않고 API visibility와 project-scoped authorization을 닫았다. | 늦게 닫힌 authorization decision도 기존 UX contract를 흔들지 않고 넣을 수 있음을 확인했다. |
| Review feedback이 guard로 환원됐다 | 6.3, 6.5, 6.9에서 review finding이 단발 수정이 아니라 regression guard로 남았다. | code review 결과는 문구 수정만이 아니라 contract test와 negative test로 남긴다. |

## Review feedback / hardening pattern

Epic 6의 review feedback은 대부분 같은 패턴으로 수렴했다.

| Pattern | Epic 6에서의 적용 |
|---|---|
| Link/source validation first | Project applications link, dashboard link, evidence link, snapshotTrend link, snapshot detail link는 selected context와 내부 path shape를 확인한 뒤에만 fetch했다. |
| Response identity fail-closed | Project/Application/Instance/Snapshot response가 선택된 context와 맞지 않으면 일부 렌더링 대신 generic safe state로 수렴했다. |
| Malformed payload fail-closed | application item nested shape, instance evidence nested block, trend point, history/marker/detail source shape는 부분 렌더링보다 safe error를 택했다. |
| Stale response guard | overlapping request와 token clear, Project 재선택, cached back action에서 이전 response가 새 화면을 덮지 않게 했다. |
| Copy hardening | `metric_data_idle`, `telemetry_unreachable`, `observing_recovery`, empty event/marker/detail copy가 정상/장애/복구 완료를 단정하지 않게 했다. |
| Security surface guard | browser token persistence, URL token parsing, cookie session, provider token/raw payload/secret exposure를 만들지 않았다. |
| Product scope guard | public project creation, Create Project UI, invite/admin/billing, Alert/Discord surface, SPA/frontend build stack을 열지 않았다. |
| Source-of-truth guard | lifecycle state, starter diagnosis, zeroInsight, recovery, rule, p95/p99, endpoint priority, marker/event promotion은 frontend에서 재계산하지 않았다. |

구체적인 review hardening 예시는 아래와 같다.

- Story 6.3: project mismatch response, invalid applications link exposure, malformed payload, stale Application reset, nested shape validation을 보강했다.
- Story 6.5: evidence `401`, malformed nested evidence, suppressed endpoint evidence, invalid snapshotTrend handoff, percentile point membership guard를 보강했다.
- Story 6.9: `metric_data_idle` guidance가 stale/down 단정처럼 읽힐 수 있는 P2 finding을 중립 copy와 focused guard로 해결했다.

## 남은 리스크와 open decisions

| 항목 | 현재 상태 | 후속 판단 |
|---|---|---|
| Project creation과 key lifecycle | Project visibility와 resource authorization은 membership으로 닫혔지만, project creation 방식, ownership, project key 발급/회전/재발급은 열지 않았다. | public onboarding API, local/internal seed, owner/role/key workflow를 별도 decision으로 다룬다. |
| Browser token persistence | Epic 6 static UI는 in-memory Bearer token hook만 사용했고 URL/cookie/storage persistence를 만들지 않았다. | logout/session UX와 token storage threat model을 먼저 닫는다. |
| Frontend architecture | Epic 6은 static HTML/CSS/JS asset으로 MVP 화면 흐름을 닫았다. | React/Vite/TypeScript/SPA 전환은 Post-MVP frontend architecture decision으로 다룬다. |
| Alert/Discord surface | Snapshot/history는 bounded operational history이며 alert delivery log와 섞지 않았다. | Alert/Discord, incident workflow, acknowledgement, owner/comment 기능은 별도 surface로 설계한다. |
| Post-MVP runtime aggregate | MVP는 latest scalar/runtime hint와 server read model 표시만 유지했다. | `latest/max/avg/sampleCount` aggregate는 schema/API/read model contract를 함께 갱신한 뒤 후보 story로 다룬다. |
| Demo seed와 운영 onboarding | Demo는 fixture와 membership guard로 검증했고 public onboarding API를 열지 않았다. | 운영용 project bootstrap과 demo seed 전략은 project creation decision과 함께 정리한다. |
| Copy drift | Story 6.9에서 주요 copy guard를 강화했지만 화면이 늘어나면 과잉 단정 표현이 다시 들어올 수 있다. | copy guard checklist를 future UI story의 acceptance에 넣는다. |
