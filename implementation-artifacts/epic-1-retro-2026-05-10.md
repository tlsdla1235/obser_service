# Epic 1 Retrospective - Architecture Foundation

## 1. 회고 범위

이 문서는 Epic 1 전체 완료 회고가 아니라, 현재 완료된 Portal MVC foundation 범위에 대한 부분 회고다.

`implementation-artifacts/sprint-status.yaml` 기준으로 Epic 1은 아직 `in-progress`이며, `1-1-starter-package-skeleton`은 `backlog`로 남아 있다. 완료된 story는 다음 세 개다.

| Story | 상태 | 핵심 산출물 |
|---|---|---|
| 1.2 Portal MVC Package Skeleton | done | Gradle Groovy DSL root build, `observability-portal` module, feature-first MVC package skeleton, smoke test |

| Story | 상태 | 영향 |

## 7. 다음 Epic 준비 상태

Epic 2는 Starter Direct Ingest Producer다. 목표는 host app request path를 막지 않고 bucket을 전송하는 starter 기능을 만드는 것이다.

Epic 2가 Epic 1 결과에 의존하는 부분:

| 의존성 | 현재 상태 | 판단 |
|---|---|---|
| Portal MVC module | 준비됨 | Epic 3/5 portal 구현으로 확장 가능 |
| Portal package boundary guard | 준비됨 | 이후 portal controller/service/repository 작업에 유효 |
| Catalog physical schema foundation | 준비됨 | Epic 3 ingest acceptance에서 확장 가능 |
| Starter module/package skeleton | 미완료 | Epic 2 착수 전 보완 필요 |
| Starter non-blocking guard | 미완료 | Epic 2 story에 포함하거나 별도 선행 story로 분리 필요 |

Winston (System Architect): "중요한 발견은 하나입니다. Epic 2는 starter 작업인데 starter module 자체가 아직 없습니다. 이건 단순 잔여 작업이 아니라 다음 Epic 진입 조건입니다."

John (Product Manager): "제품 방향은 바꾸지 않아도 됩니다. 다만 Epic 2 story 순서는 현재 현실을 반영해야 합니다."

Amelia (Developer): "따라서 Epic 2를 현재 backlog 그대로 시작하기보다, starter bootstrap을 먼저 닫는 작업을 선행 조건으로 두는 것이 안전합니다."

## 10. Team Agreements

- Epic 상태를 말할 때는 "전체 Epic 완료"와 "현재 구현 범위 완료"를 구분한다.
- foundation story에서는 범위 확장을 좋은 의도라도 바로 끌어오지 않는다.
- architecture guard는 구현을 방해하는 규칙이 아니라, 다음 구현자가 안심하고 확장할 수 있게 만드는 안전장치로 다룬다.
- starter 작업을 시작하기 전 module skeleton, package boundary, request path non-blocking guard의 최소 기준을 먼저 닫는다.
- story 기록에는 실패 원인, 최종 검증 명령, 의도적으로 제외한 범위를 계속 남긴다.
