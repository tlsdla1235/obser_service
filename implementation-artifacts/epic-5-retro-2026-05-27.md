### 1.8 5.8과 5.9의 pre-story contract decision 효과

5.8과 5.9는 구현 전에 contract decision을 먼저 잠근 것이 구현 안정성에 직접적인 영향을 줬다.

5.8은 하나의 큰 "snapshot persistence and marker" story로 남겼다면 writer, detail, marker, recovery, event handoff가 한 번에 섞일 가능성이 컸다. decomposition decision이 5-8-a와 5-8-b를 나누면서 "무엇을 언제 저장하는가"와 "저장된 것을 어떻게 읽는가"가 분리됐다. 그 결과 writer는 capture reason, duplicate identity, helper column, bounded endpoint evidence, bounded instance summary에 집중했고, reader는 current 재판정 금지, previousState, lastHealthyAt, recovery marker, endpoint anchor에 집중할 수 있었다.

5.9도 같은 효과가 있었다. pre-story contract가 marker와 event를 분리하고, event source를 `dashboard_snapshots` row metadata/helper/stored JSON으로 제한했기 때문에 5.9-a는 source boundary와 API skeleton을 안전하게 만들 수 있었고, 5.9-b는 promotion, suppression, period folding에만 집중할 수 있었다. 특히 `markerId`를 `eventId`로 재사용하지 않는 결정, marker severity를 event severity로 강제하지 않는 결정, 60분 suppression을 writer upsert와 분리한 결정은 후반 구현의 혼선을 줄였다.

이 방식은 Epic 6에서도 반복할 가치가 있다. UI story 중 하나라도 "화면", "copy", "API fallback", "demo fixture"가 동시에 섞이면 먼저 contract decision으로 source, non-goal, copy guard, test guard를 잠그는 편이 안전하다.

### 1.10 Closure 판단

Epic 5는 closure 가능하다.

닫힌 기준:

- Application Dashboard가 server-computed read model/API로 현재 상태와 next action을 제공한다.
- Project/Application navigation은 scope 선택과 light summary에 머문다.
- Instance Evidence는 application 판단을 대체하지 않는 drill-down이다.
- Instance Snapshot Trend는 stored dashboard snapshot/read model의 bounded instance summary projection이다.
- Snapshot detail/marker는 저장 당시 read model을 보여주고 current state를 재판정하지 않는다.
- Operational Event History는 stored snapshot/read model 후보를 bounded event feed로 접고, 별도 event store를 만들지 않는다.

Epic 5를 닫는 의미는 UI가 완성됐다는 뜻이 아니다. Epic 6이 계산 없이 렌더링하고 demo로 검증할 수 있는 API/read model 계약이 충분히 닫혔다는 뜻이다.

## 2. Next Epic Preparation

### 2.1 Epic 6 진입 해석

Epic 6은 Epic 5 read model/API를 소비하는 user-facing UX와 demo hardening epic이다. 핵심은 화면이 새 판단 engine이 되는 것이 아니라, 사용자가 실제로 밟는 `project -> application -> dashboard -> instance evidence -> instance trend -> snapshot/history` 흐름을 끊기지 않게 연결하는 것이다.

Epic 6의 성공 조건은 아래처럼 잡는다.

- Project Entry는 scope 선택과 setup guide 진입 화면이다.
- Application List는 상태 스캔과 dashboard 진입을 돕지만 상세 판단을 대체하지 않는다.
- Application Dashboard는 primary first-screen이며 Epic 5 dashboard read model을 그대로 렌더링한다.
- Instance Detail은 evidence drill-down이고 application 판단을 대체하지 않는다.
- Instance Snapshot Trend와 Snapshot/History는 stored dashboard snapshot/read model projection이다.
- Demo는 heartbeat, first accepted bucket, insufficient sample, no-triage baseline, failure/recovery path를 사용자가 이해할 수 있게 보여준다.

### 2.2 Epic 6 source-of-truth 순서

Epic 6 story 작성과 구현은 아래 순서를 따른다.

1. `planning-artifacts/source-of-truth/current-product-source-of-truth.md`
2. 최신 `planning-artifacts/contracts/*`
3. `planning-artifacts/prototypes/epic5-6-dashboard-flow-prototype.html`
4. `planning-artifacts/epics.md`
5. `planning-artifacts/sprint-plan.md`
6. Epic 5 story/spec handoff
7. `implementation-artifacts/sprint-status.yaml`

충돌이 생기면 최신 product source-of-truth와 contract를 우선한다. 과거 문서가 application-only first screen, alert-first surface, raw explorer, UI-side calculation을 전제하면 Epic 6 기준으로 정렬 대상이다.

### 2.3 Epic 6에서 반드시 유지할 계약

| 계약 | Epic 6 적용 |
|---|---|
| UI는 server read model을 표시한다 | lifecycle state, starter diagnosis, zeroInsight, recovery, rule, p95/p99, endpoint priority, marker/event를 frontend에서 재계산하지 않는다. |
| Application Dashboard가 primary first-screen이다 | Project와 Application List는 scope 선택/스캔 화면이고 상세 판단은 dashboard에서 본다. |
| Metric data axis와 starter connection axis를 분리한다 | State strip과 starter connection strip을 시각적으로도 분리한다. 최근 heartbeat와 오래된 accepted bucket을 host down으로 합치지 않는다. |
| Percentile은 source-scoped starter canonical point다 | UI label은 source/scope를 드러내고, 여러 instance/window p95/p99 평균/최댓값/병합을 만들지 않는다. |
| Histogram은 distribution/evidence다 | histogram bucket으로 p95/p99를 계산하지 않는다. |
| Endpoint priority는 next-check surface다 | root cause 확정 순위나 endpoint p99 alert처럼 표현하지 않는다. |
| Instance Detail은 evidence drill-down이다 | instance health score, instance lifecycle state 재판정, application state 대체 판단을 만들지 않는다. |
| Trend/history는 stored read model projection이다 | raw bucket explorer, raw snapshot JSON explorer, arbitrary time-series query로 확장하지 않는다. |
| Snapshot detail은 저장 당시 read model이다 | current dashboard fallback으로 expired/missing snapshot detail을 보완하지 않는다. |
| Operational event history는 bounded feed다 | `operational_events` table 없이 stored snapshot/read model을 접은 결과이며, alert delivery log와 섞지 않는다. |
| Empty/missing copy는 상태를 단정하지 않는다 | `events=[]`, `markers=[]`, expired detail, missing trend point를 "현재 문제 없음", "복구 완료", "앱 정상 확정"으로 표현하지 않는다. |

### 2.7 Epic 6 readiness checklist

- `DashboardReadModelService` response를 UI source-of-truth로 사용한다.
- Project Entry와 Application List는 dashboard 판단을 대체하지 않는다.
- Dashboard UI는 `zeroInsight`, `recovery`, `triageCards`, `endpointPriority`, `sourceScopedPercentiles`, `histogramDistribution`, `instances`, `snapshot`을 서버 응답 그대로 표시한다.
- Instance Evidence UI는 accepted bucket freshness와 starter heartbeat를 같은 health로 합치지 않는다.
- Instance Snapshot Trend UI는 stored `instanceSummary.items[]` projection을 표시하고, trend point로 instance health score를 만들지 않는다.
- Snapshot detail UI는 stored detail이며 current state recalculation이 없다는 의미를 유지한다.
- Marker UI와 Operational Event History UI는 같은 snapshot deep link를 공유할 수 있지만 같은 contract로 합치지 않는다.
- Empty states는 source absence를 말하고 current health를 단정하지 않는다.
- No raw explorer, no arbitrary query, no endpoint timeseries, no endpoint p95/p99, no `operational_events` table 기준을 유지한다.

### 2.8 Epic 6 진입 결론

Epic 6 진입을 막는 중대한 방향 전환은 없다. Epic 5는 UI가 소비할 server-computed read model/API 계약을 충분히 닫았다.

다만 Epic 6의 위험은 기능 부족보다 "UI가 친절해지려다가 계산을 되가져오는 것"에 있다. Epic 6은 새 판단을 만드는 epic이 아니라, Epic 5가 만든 판단을 사용자가 이해할 수 있는 흐름과 copy로 연결하는 epic으로 운영해야 한다.
