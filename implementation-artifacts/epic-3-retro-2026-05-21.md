## 3. Epic 3 목표 달성 여부

결론: Epic 3는 closure 가능하다. 단, duplicate/idempotency closure 기준은 기존 full idempotency 문구가 아니라 `mvp-deferred-risk-spec.md`와 Story 3.4 완료 기록에 정렬된 MVP 정책이다.

| 목표 | 판단 | 근거 |
|---|---|---|
| Project key verification | 달성 | Story 3.1은 missing/blank, malformed, BCrypt mismatch, disabled project를 unauthorized로 닫고 active project만 `VerifiedProject`로 반환한다. raw key가 DB, log, response, repository lookup surface에 남지 않도록 guard를 추가했다. |
| Portal ingest validation | 달성 | Story 3.2는 Story 2.5 golden envelope를 success fixture로 사용하고, `schemaVersion: 1.0`, UTC 30초 bucket, metric taxonomy, route/idempotency key consistency를 service layer에서 검증한다. |
| `accepted_metric_buckets` persistence | 달성 | Story 3.3은 Flyway V003, Korean comments, JPA repository, catalog get-or-create, first accepted ingest `201 Created` path를 구현했다. `dashboard_snapshots`, `operational_events`, view, trigger, stored procedure는 추가하지 않았다. |
| MVP duplicate/idempotency policy | 달성 | Story 3.4는 같은 `(project_id, idempotency_key)` 재수신을 payload hash 비교 없이 `409 duplicate_idempotency_key`로 reject하고, 기존 row overwrite/second row 생성을 막는 검증을 추가했다. |
| Epic 3 scope guard | 달성 | Story 기록과 architecture guard가 dashboard/read-model/lifecycle/insight/history 계산을 Epic 3 밖으로 유지했다. |

### 중요한 closure 해석

Epic 3는 "full idempotent replay success"가 아니라 "MVP duplicate key reject" 정책으로 닫는다. `ingest-envelope.md`, `sprint-plan.md`, Story 3.4 초기 AC에는 같은 payload retry를 `200 OK duplicate=true`로 수렴시키는 문구가 남아 있으나, active MVP 결정은 `mvp-deferred-risk-spec.md`와 `deferred-work.md`가 우선한다.

이 차이는 구현 실패가 아니라 scope control 결정이다. 다만 후속 Epic에서 혼선이 생기지 않도록 문서 정리 action item으로 남긴다.

## 7. Deferred / Residual Risk 정리

| 항목 | 현재 상태 | 남은 위험 | 재검토 지점 |
|---|---|---|---|
| Full idempotent replay success | MVP 밖 | 저장 성공 뒤 응답 손실이 발생하면 retry가 success로 수렴하지 않는다. | starter retry UX를 제품 품질로 보장하거나 external producer를 허용할 때 |
| Payload hash 기반 conflict 분류 | MVP 밖 | same payload retry와 different payload conflict를 진단상 구분하기 어렵다. | full idempotency story를 다시 열 때 |
| Insert race convergence | MVP 밖 | 동시 요청이 duplicate/conflict 대신 persistence failure처럼 보일 수 있다. | concurrency/load 검증을 도입할 때 |
| Catalog get-or-create race hardening | MVP 밖 | 같은 application/instance 최초 ingest가 동시에 들어오면 catalog unique violation이 먼저 드러날 수 있다. | parallel ingest path를 검증할 때 |
| Strict JSON token type | residual | `schemaVersion: "1.0"`을 유지한 malformed scalar가 Jackson coercion으로 DTO에 들어올 수 있다. | external collector/third-party SDK를 허용할 때 |
| Endpoint cap / duplicate endpoint key | deferred | 과도한 endpoint JSON 또는 같은 `method + route` 중복이 read model merge 품질에 영향을 줄 수 있다. | p95/read model merge 구현 전 |
| Histogram boundary set cross-check | deferred | summary와 endpoint boundary set 차이가 p95 merge 정확도를 흔들 수 있다. | Histogram merge 구현 전 |
| DB/JPA deep validation | deferred | service를 우회하는 future import/admin path가 invalid row를 넣을 수 있다. | ingest service 외 저장 경로를 만들 때 |
| Index/test hardening | deferred | query plan tuning과 cleanup 성능 기준을 아직 닫지 않았다. | retention cleanup 또는 query plan 최적화 전 |

## 9. Next Epic Preparation

Epic 4는 Epic 3의 저장 결과를 계산 입력으로 사용한다. 준비 판단은 "data is accepted"에서 "state language is trustworthy"로 초점이 바뀌는 것이다.

준비 체크:

- `accepted_metric_buckets` query surface가 state window 계산에 필요한 최소 조회를 제공하는지 확인한다.
- state semantics는 accepted bucket 저장 시점이 아니라 bucket end/freshness 기준으로 판단한다.
- Epic 4는 dashboard snapshot/history를 만들지 않는다. snapshot/history는 Epic 5/6 경계로 유지한다.
- Epic 5 시작 전에는 endpoint/histogram deferred 항목을 다시 열어 p95 merge 품질을 닫는다.
- Epic 6 시작 전에는 full idempotency UX와 onboarding/demo 문구를 다시 확인한다.

Significant discovery: Epic 4/5/6 계획을 중단시킬 방향 전환은 없다. 다만 duplicate/idempotency 문구 drift는 후속 planning confusion을 만들 수 있으므로 문서 정리 action item으로 처리해야 한다.

## 10. Epic 3 Closure 판단

Closure 판단: 승인 가능.

조건:

- Epic 3의 완료 기준은 active MVP duplicate/idempotency policy로 해석한다.
- Story 3.1-3.4의 `done` 상태와 test evidence를 closure 근거로 삼는다.
- full idempotent replay success, payload hash conflict classification, race convergence는 deferred/residual risk로 추적한다.
- dashboard read model, lifecycle state, insight rules, operational event history는 Epic 3 closure 대상이 아니다.

Epic 3를 닫아도 되는 이유:

- project authorization boundary가 닫혔다.
- portal-side validation boundary가 닫혔다.
- accepted bucket persistence와 catalog get-or-create path가 닫혔다.
- duplicate key 기반 MVP no-second-row policy가 닫혔다.
- 후속 Epic으로 넘길 항목이 기능 누락이 아니라 명시된 deferred risk로 분리됐다.

## 12. Team Agreements

- Epic 3 closure는 full exactly-once나 full idempotent replay success를 의미하지 않는다.
- duplicate handling은 starter duplicate flush primary mechanism이 아니라 portal retry safety boundary다.
- accepted bucket 저장소는 read model의 입력이지 read model 자체가 아니다.
- state, p95, insight, endpoint priority, operational event history 계산은 후속 Epic service layer에서 닫는다.
- deferred risk는 숨기지 않고 후속 Epic의 entry checklist로 재검토한다.
