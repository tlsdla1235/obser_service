## 3. Epic Review

Epic 2의 목표는 starter를 추가한 host app request path를 막지 않고 30초 bucket을 portal ingest API로 보낼 준비를 하는 것이었다.

완료된 producer path:

1. Micrometer/Spring observation signal을 starter 내부 input으로 변환한다.
2. raw path와 high-cardinality label을 low-cardinality route/tag 경계에서 차단한다.
3. normalized route 기준으로 app/endpoint bucket을 집계한다.
4. grace 이후 drain 대상 bucket을 bounded queue에 넣고 background worker에서만 flush한다.
5. sealed bucket을 `ingest-envelope` contract의 payload와 idempotency header 후보로 변환한다.
6. Prometheus scrape, pull query, arbitrary query UI, p95/read-model 계산이 starter MVP path에 들어오지 않도록 negative guard로 고정한다.

John (Product Manager): "제품 방향은 유지됐습니다. Epic 2는 화면을 만들지 않았지만, 첫 화면이 믿을 수 있는 데이터를 받기 위한 producer 쪽 약속을 닫았습니다."

Dana (QA Engineer): "특히 좋았던 점은 happy path만 본 게 아니라, portal down, queue overflow, duplicate candidate, query leakage, forbidden dependency를 테스트로 고정했다는 점입니다."

## 8. Epic 3 준비 상태

Epic 3 목표는 portal이 ingest envelope를 검증하고 idempotent하게 저장하는 것이다. Epic 2 결과에 직접 의존하는 부분은 아래와 같다.

| Epic 3 의존성 | Epic 2 제공 상태 | 준비 판단 |
|---|---|---|
| `X-OBS-Project-Key` header | Story 2.5가 local identity와 header 후보를 만든다. | portal 검증 구현 필요 |
| `schemaVersion: 1.0` payload | Story 2.5 golden JSON contract가 있다. | 수용 validation fixture로 재사용 가능 |
| UTC 30초 bucket metadata | Story 2.3/2.5가 boundary와 serialization을 고정했다. | portal boundary validation 필요 |
| normalized route-only endpoint | Story 2.2/2.6이 raw path/query/high-cardinality 차단을 고정했다. | Story 2.2 status closure 완료 |
| idempotency key | Story 2.5가 deterministic key 후보를 만든다. | duplicate success/conflict 구현 필요 |
| `accepted_metric_buckets` persistence | Epic 2 scope 밖으로 유지됐다. | Epic 3 repository story에서 구현 |

Significant discovery: Epic 3 계획 자체를 바꿔야 할 정도의 방향 전환은 없다. Story 2.2 상태 불일치는 closure 과정에서 해소됐고, portal-side validation mirror는 Epic 3의 핵심 선행 설계 포인트로 유지한다.

## 10. Team Agreements

- Epic 3에서는 starter가 보낸 payload를 신뢰하지 않고 portal `IngestAcceptanceService`에서 다시 검증한다.
- `schemaVersion: 1.0`에는 latest-only JVM/datasource ratio만 허용하고 Post-MVP aggregate 후보를 섞지 않는다.
- raw path candidate, query string, query key/value, high-cardinality tag, free-form tag map, arbitrary custom metric map은 portal 저장 경로에서도 거부한다.
- duplicate handling은 idempotent acceptance safety net으로 구현하되, starter duplicate flush primary mechanism으로 해석하지 않는다.
- review closure는 code/test 통과와 story/sprint-status 정합성을 함께 포함한다.
